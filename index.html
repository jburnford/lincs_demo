A proof of concept

Following Indian Agents across the archive

What becomes possible when 20 million pages of historical Canadian government records have clean OCR, named-entity recognition, and linked-open-data grounding.

Corpus: Canada Department of Indian Affairs Annual Reports, 1880–1885 · Linked to the LINCS Historical Indian Affairs Agents graph

The problem with keyword search

A historian researching nineteenth-century Indian Affairs in Canada today can search Canadiana.ca for a name — say, Dewdney — and receive a flat list of OCR hits. The historian must then read each result to figure out which Dewdney, in which year, in which place, in which administrative role. The same person appears as E. Dewdney, Hon. Edgar Dewdney, and The Commissioner; the same role is described in dozens of different phrasings; the same place is named under English, French, and Indigenous spellings.

Linked Open Data fixes this. Once each person, place, and role mentioned in the reports is grounded to a stable URI in an authority like LINCS, Wikidata, VIAF, or GeoNames, a historian can ask questions that keyword search cannot answer at all.

This page demonstrates the payoff using six volumes of the Annual Reports of the Department of Indian Affairs (1880–1885), processed with NER and grounded against the LINCS Historical Indian Affairs Agents graph. Of the 355 agents LINCS records as active in this window, 174 (49%) appear in our reports — a strong basis for cross-reference.

Regional discourse at scale

A quantitative payoff that keyword search cannot replicate.

We partition the 1880–1885 sections into three regional corpora — British Columbia, Prairies / North-West Territories, and Quebec — based not on where the text says it is, but on the LINCS-grounded place of the agents mentioned in each section. We then run TF-IDF across the three corpora and surface the most distinctive vocabulary in each.

Charts load from data/regional_tfidf.json

These three corpora were assembled without reading any text — purely by following LOD links from agent names to agency places. Keyword-based partitioning could not build these corpora correctly: it would miss paragraphs that name the agent without naming the region, and falsely include paragraphs where the region is mentioned in passing. The method is corpus-size-invariant: what you see on 500 sections runs unchanged on 20 million.

How this was built

  1. OCR via OLMoCR on the Plato HPC cluster (A100 GPUs).
  2. Section parsing of the resulting markdown into sub-agency reports, each ~1–3 pages of narrative.
  3. NER with a local 26B-parameter language model (Gemma 4 26B-A4B-it via vLLM), extracting persons, indigenous groups, places, organisations, events, and relationships per section.
  4. Grounding by SPARQL against the LINCS ind-affairs graph, matching person mentions to E21_Person URIs by surname, initial, and year-of-occupation overlap.
  5. Turtle generation following the LINCS application profile: each report section becomes an E73_Information_Object with P67_refers_to pointers to the grounded agent and place URIs.
  6. Visualisation as this static page, which fetches a pre-computed JSON payload for offline reliability and uses YASGUI for live queries against LINCS Fuseki.

An example fragment of generated Turtle

@prefix crm: <http://www.cidoc-crm.org/cidoc-crm/> .
@prefix viaf: <http://viaf.org/viaf/> .
@prefix dia:  <http://example.org/dia/> .

dia:section/1882_dewdney_nwt a crm:E73_Information_Object ;
    rdfs:label "1882 NWT report by Edgar Dewdney"@en ;
    crm:P129_is_about viaf:20583753 ;
    crm:P67_refers_to viaf:20583753, geonames:6141242/ ;
    crm:P190_has_symbolic_content "…the Commissioner visited the Blackfoot…"^^xsd:string .
